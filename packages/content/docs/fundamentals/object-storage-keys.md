## 一句话结论

对象存储通常不是传统文件系统。它的基本组织方式是：

```plaintext
bucket + object key -> object data + metadata
```

我们在控制台里看到的“目录”，大多只是对象 key 里的一段普通字符串。比如下面这三个对象：

```plaintext
photos/2026/06/cat.jpg
photos/2026/06/dog.jpg
photos/2026/notes.txt
```

它们看起来像是位于 `photos/2026/06/` 目录下，但对象存储真正保存的是三个完整 key。`/` 通常只是 key 名称中的一个字符，控制台、SDK 或网关把它解释成目录层级，方便人阅读和筛选。

## 和文件系统的核心差别

在本地文件系统中，目录通常是一个真实的元数据节点。你可以创建空目录，可以对目录做重命名，也可以把目录当成一个层级边界来理解权限、遍历和移动。

对象存储里更常见的模型是扁平命名空间。bucket 下面放着很多对象，每个对象有自己的 key、内容、metadata、版本信息和存储类型。所谓“目录”通常是通过 key 前缀模拟出来的。

这会带来几个直接差异：

1. 创建目录不一定真的创建了目录对象。
2. 空目录可能不存在，因为没有任何对象以这个前缀开头。
3. 重命名目录通常不是一次元数据操作，而是批量复制新 key 再删除旧 key。
4. 移动目录一般也不是原子操作，中途失败时可能出现新旧 key 同时存在的状态。
5. 列出目录本质上是按前缀查询对象列表，而不是读取某个目录节点。

所以，在对象存储里问“这个目录是否存在”，更准确的问题往往是：

```plaintext
是否存在至少一个以 photos/2026/06/ 为前缀的对象？
```

## prefix 和 delimiter

对象存储服务通常会提供按前缀列举对象的能力。假设 bucket 中有这些 key：

```plaintext
photos/2026/06/cat.jpg
photos/2026/06/dog.jpg
photos/2026/07/bird.jpg
photos/raw/cat.nef
readme.txt
```

如果按 `photos/2026/` 这个 prefix 列举，就可以得到：

```plaintext
photos/2026/06/cat.jpg
photos/2026/06/dog.jpg
photos/2026/07/bird.jpg
```

如果再指定 `/` 作为 delimiter，服务或 SDK 可能会把下一层共同前缀折叠成类似目录的结果：

```plaintext
photos/2026/06/
photos/2026/07/
```

这里的 `06/` 和 `07/` 不是文件系统目录，而是从 key 字符串中归纳出来的 common prefix。不同对象存储服务的 API 名称会有差异，但思路基本相同：prefix 用来筛选，delimiter 用来把扁平 key 模拟成层级视图。

## “文件夹对象”是什么

有些控制台允许点击“新建文件夹”。这类操作经常会创建一个特殊对象，例如：

```plaintext
photos/2026/06/
```

它的 key 以 `/` 结尾，内容长度可能是 0，metadata 里可能带着特定标记。这个对象的存在可以让控制台显示一个空文件夹，但它仍然只是一个普通对象，不是传统文件系统里的目录节点。

因此，应用逻辑不要强依赖“文件夹对象”。如果业务关心某个集合是否存在，通常应该用数据库、业务索引或对象列表来表达，而不是依赖一个空的占位 key。

## key 设计会影响后续维护

对象 key 一旦进入生产环境，就会影响访问路径、生命周期规则、权限策略、批处理脚本、缓存和迁移成本。命名时可以先考虑这几个问题：

1. 查询最常用的维度是什么。
2. 是否需要按租户、业务类型、日期或资源类型分组。
3. 是否需要批量删除或批量归档某一类对象。
4. 是否需要避免单个前缀下对象数量过大导致列举成本上升。
5. 是否会把 key 暴露在 URL、日志或用户可见界面里。

一个常见的业务 key 可能长这样：

```plaintext
tenant-a/invoices/2026/06/receipt-0001.pdf
tenant-a/invoices/2026/06/receipt-0002.pdf
tenant-b/avatar/user-123.png
```

这种设计让人一眼能看出归属和用途，也方便按 `tenant-a/invoices/2026/06/` 做批量处理。但是它也意味着“把 2026 年 6 月发票移动到另一个路径”会变成大量对象的复制和删除。

如果对象访问非常高频，还需要查阅所用服务的性能建议。S3 这类成熟服务已经不再要求人为给 key 加随机前缀来分散分区压力，但一些 S3 兼容实现、私有对象存储或旧系统可能仍有自己的热点前缀限制。不要把某一家服务的历史经验直接套到所有对象存储上。

## 常见误区

### 误区一：删除目录就是删除一个目录节点

在对象存储中，删除“目录”通常意味着删除所有以某个 prefix 开头的对象。比如删除 `photos/2026/06/`，实际要删除的是：

```plaintext
photos/2026/06/cat.jpg
photos/2026/06/dog.jpg
photos/2026/06/
```

如果中途失败，可能只删掉一部分对象。批量删除脚本应该支持分页、重试、断点续跑和结果校验。

### 误区二：改名只是改 metadata

把 `photos/2026/06/` 改成 `photos/2026/june/`，通常要为每个对象生成新 key：

```plaintext
photos/2026/06/cat.jpg  -> photos/2026/june/cat.jpg
photos/2026/06/dog.jpg  -> photos/2026/june/dog.jpg
```

这个过程可能产生复制费用、请求费用和短暂的不一致状态。对象数量越多，迁移越应该先做清单、灰度、校验和回滚设计。

### 误区三：可以像文件一样原地追加

对象更像一个完整 blob，而不是可随机写入的普通文件。很多对象存储支持 multipart upload，用来分片上传大对象，但这不等于可以像本地文件一样随意 seek、append、truncate。日志、数据库文件、频繁小块更新的数据，通常不适合直接把对象存储当成本地磁盘使用。

### 误区四：路径权限自然继承

文件系统目录常有继承权限的直觉，但对象存储的权限可能来自 bucket policy、IAM、ACL、临时签名、网关规则或应用自己的鉴权逻辑。`private/user-a/avatar.png` 这个 key 看起来像私有目录，并不代表它天然安全。权限边界必须由明确的策略保证。

## 什么时候应该额外建索引

如果需求只是保存和按 key 获取对象，直接使用对象存储就够了。但如果需要下面这些能力，应该考虑额外维护数据库索引：

1. 按多种条件查询对象，例如用户、状态、标签、创建时间和业务类型。
2. 需要稳定分页，并且分页顺序不能受对象列举结果影响。
3. 需要事务语义，例如对象写入成功后必须同时更新业务状态。
4. 需要复杂权限判断，例如团队、角色、分享链接和过期时间。
5. 需要记录对象的业务生命周期，而不只是存储生命周期。

一种常见做法是：对象存储只负责保存内容，数据库保存对象 key、业务 ID、owner、checksum、content type、size、状态和创建时间。业务查询先查数据库，真正下载时再用 key 去对象存储读取。

## 实用检查清单

设计对象存储 key 前，可以快速过一遍：

1. 这个 key 是否只包含稳定业务含义，避免把容易变化的展示名称放进去。
2. 是否需要把用户输入做规范化，避免空格、大小写、重复斜杠或不可见字符造成排查困难。
3. 是否能从 key 看出粗粒度归属，但不会泄露敏感信息。
4. 是否已经考虑批量删除、归档和迁移成本。
5. 是否有数据库或清单记录对象和业务实体的对应关系。
6. 是否清楚使用的对象存储在 list、overwrite、versioning、lifecycle 和权限上的具体语义。

把对象存储理解成“带 key 的对象集合”，很多问题会变得更直观：目录只是 key 的一种命名习惯，不是系统真正依赖的结构。
